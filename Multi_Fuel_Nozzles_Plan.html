Multi-Fuel Nozzles & Per-Nozzle Tank Assignment — Implementation Plan

Goals (from owner):
1. Allow a single island to dispense multiple fuel types (e.g. one nozzle diesel, one petrol — multi-product dispenser).
2. Allow each nozzle to be assigned to a specific tank, including the multi-tank case (2+ diesel tanks, 2+ petrol tanks).
3. Owner shouldn't have to fiddle nozzle-by-nozzle — sensible presets per island, with picking only when ambiguous.
Constraint: existing functionality, outputs, reports, and reconciliation numbers must remain 100% correct. No silent drift.
Owner decisions — all locked in: 8 design questions resolved (Section 14 below). Notable consequences baked into this plan:
  • Strict fuel/tank validation (no force=true escape) — changing a nozzle's fuel requires changing its tank in the same atomic call.
  • Tank assignment is treated as permanent configuration — no transfer-event machinery; reassignment is a rare admin action accepted with manual reconciliation if it ever happens.
  • Per-nozzle row layout for mixed islands on the shift assignment grid (each nozzle is its own coloured row inside the island card).
  • Per-fuel label namespaces (LSD 1A + UNL 1A coexist), Island.product_type auto-computed, query-time recon, Phase 2 retirement of PumpStation.tank_id, Custom mode behind an Advanced expander.

1. Executive Summary

The fuel station model today assumes one island = one fuel. The plan upgrades that to any island can mix fuels at the nozzle level, and each nozzle can be wired to a specific tank. The change is delivered as preset templates (All Diesel, All Petrol, Mixed) so 95% of stations never see a tank picker, while multi-tank stations get a small dropdown only when there's genuine ambiguity.

Why this is low-risk: the codebase is already 80% ready. Nozzle.fuel_type is already per-nozzle, get_tank_id_for_nozzle() already exists with a nozzle-first resolution path, sales records already carry tank_id frozen at sale time, reconciliation already uses a three-tier fallback (sale.tank_id → nozzle→tank → fuel_type). The refactor mostly activates what's already there.

Backward compatibility guarantee: for every existing station that does not deliberately configure a mixed-fuel island after upgrade, all outputs are byte-identical to today: same revenue numbers, same CSV/Excel rows, same reconciliation deltas, same labels. This is verified by the 42-scenario regression suite (Section 10).

2. Design Decisions

D1 — Move tank_id authority from PumpStation to Nozzle

DECIDED Add Nozzle.tank_id; demote PumpStation.tank_id to a fallback default that's only consulted when a nozzle has no explicit tank.

Why: a multi-product dispenser is physically one pump head with two nozzles drawing from two tanks. Modelling this as one PumpStation with mixed nozzles is faithful; modelling it as two PumpStations would double every report row for an MPD.

Why not the alternative (multiple pump stations per island): ripples into every place that does island.pump_station (20+ call sites). Strictly more code churn for a worse physical model.

D2 — Preset-first UI

DECIDED Owner picks a preset per island (All Diesel / All Petrol / Mixed), with a "Custom..." escape hatch for edge cases. Backend wires nozzles to tanks automatically.

Why: mirrors how owners think ("island 3 is the mixed pump"); impossible to wire a petrol nozzle to a diesel tank via presets; collapses 3+ form fields into one choice.

D3 — Tank picker shown only on ambiguity

DECIDED If a fuel has only one matching tank: auto-wire, no UI. If multiple tanks of that fuel exist: one dropdown per fuel slot in the chosen preset, filtered to matching fuel only.

D4 — Nozzle fuel type strictly tied to tank

DECIDED nozzle.fuel_type remains as a stored field (for code paths that already read it) but is enforced to equal tanks[nozzle.tank_id].fuel_type. Backend always rejects mismatched assignments with HTTP 400 — no force escape hatch.

Consequence: changing a nozzle's fuel requires changing its tank in the same atomic call. The PUT /islands/{id}/nozzle/{id}/fuel-type endpoint requires both fuel_type and tank_id in its body. You cannot ever observe a nozzle whose fuel disagrees with its tank.

Why "stored not derived": ~6 backend modules and ~5 frontend pages already read nozzle.fuel_type; making it derived means changing every read site. Storing it (and validating on write) is a smaller, safer change.

D5 — Per-fuel label namespaces

DECIDED Numbering becomes per-fuel-within-island. A mixed island (one diesel + one petrol nozzle) produces labels like LSD 1A and UNL 1A on the same physical island — no collision because the namespaces are separate. This preserves the spreadsheet mental model owners use today.

D6 — Island.product_type becomes informational

DECIDED Field stays in the model as an optional label hint. Computed automatically: "Petrol" / "Diesel" / "Mixed" / null, based on the actual nozzle composition. Never used as a gate for sale routing or filtering — every consumer walks nozzles instead.

D7 — PumpStation.tank_id retained as fallback

DECIDED Field becomes optional; resolution order is nozzle.tank_id → pump.tank_id → fuel_type fallback. Slated for removal in a Phase 2 cleanup once all stations are confirmed migrated.

3. Preset System & Multi-Tank Logic

3.1 The three presets

For an island with two nozzles (positions A and B):

All Diesel

Both nozzles dispense diesel from the same diesel tank.

Writes: nozzle_A.tank_id = T_DIESEL, nozzle_B.tank_id = T_DIESEL, island.product_type = "Diesel"

All Petrol

Both nozzles dispense petrol from the same petrol tank.

Writes: nozzle_A.tank_id = T_PETROL, nozzle_B.tank_id = T_PETROL, island.product_type = "Petrol"

Mixed (MPD)

One nozzle petrol, one nozzle diesel. By convention nozzle A = petrol (left side), B = diesel.

Writes: nozzle_A.tank_id = T_PETROL, nozzle_B.tank_id = T_DIESEL, island.product_type = "Mixed"

A fourth option Custom… (hidden behind an "Advanced" expander) reveals per-nozzle tank dropdowns. Used for stations with 3+ nozzles per island, LPG dispensers later, or load-balancing across multiple tanks of the same fuel.

3.2 Multi-tank tank-picking logic

When the owner chooses a preset, the wizard inspects the station's tank inventory:

PresetStation hasUI shownDefault
All Diesel1 diesel tank(no dropdown)Auto-wired to that tank
All Diesel2+ diesel tanks"Which diesel tank? [▾]"First diesel tank by id-order
All Petrol1 petrol tank(no dropdown)Auto-wired
All Petrol2+ petrol tanks"Which petrol tank? [▾]"First petrol tank
Mixed1 of each fuel(no dropdown)Auto-wired (P→petrol, D→diesel)
Mixed2+ of either fuel"Diesel from [▾]" + "Petrol from [▾]"First of each fuel
CustomAnyPer-nozzle tank dropdownNone — owner picks each

Dropdown options are filtered to tanks of the matching fuel only. Petrol tanks never appear in a "diesel" slot — that's how the fuel-type constraint is enforced at the UI level.

3.3 Per-nozzle tank override (Custom mode)

Real but rare scenario: island 1 has both nozzles diesel, but owner wants nozzle A to draw from TANK-DIESEL-1 and nozzle B from TANK-DIESEL-2 (load balancing across tanks). The Custom preset reveals two tank dropdowns instead of one. Same applies to multi-petrol load balancing.

3.4 Backend behavior on preset application

The preset endpoint (POST /islands/{id}/preset) is a single atomic write that:

  1. Validates the preset name is one of all_diesel | all_petrol | mixed | custom
  2. For each nozzle, sets nozzle.tank_id and nozzle.fuel_type based on the preset + tank-picker output
  3. Sets pump_station.tank_id to the dominant tank (for backward compat with code paths that still consult it)
  4. Sets island.product_type to "Petrol" | "Diesel" | "Mixed"
  5. Re-runs compute_display_labels(storage) so labels reflect the new fuel composition
  6. Persists via save_station_storage
  7. Emits an audit event with the before/after diff

If any step fails, the entire write is rolled back (in-memory state restored from the deep-copy taken at step 1).

5. Migration Strategy

5.1 The migration function

def _migrate_nozzles_propagate_tank_id(storage: dict) -> int:
    """
    Copy each pump_station.tank_id down to its nozzles' tank_id field.
    Idempotent — guards against overwriting existing assignments.
    Returns the number of nozzles migrated (0 if already migrated).
    """
    migrated = 0
    skipped_no_pump_tank = 0
    for island in storage.get('islands', {}).values():
        pump = island.get('pump_station')
        if not pump:
            continue
        pump_tank_id = pump.get('tank_id')
        if not pump_tank_id:
            skipped_no_pump_tank += 1
            continue
        for nozzle in pump.get('nozzles', []):
            if nozzle.get('tank_id'):
                continue  # already migrated, leave alone
            nozzle['tank_id'] = pump_tank_id
            migrated += 1
    if skipped_no_pump_tank:
        logger.warning(
            f"[migration] Skipped {skipped_no_pump_tank} pump_stations "
            f"with no tank_id; their nozzles fall back to fuel_type heuristics"
        )
    return migrated

5.2 Trigger

Called from seed_station_defaults() as the fourth migration step, matching the pattern of the existing three. Runs lazily on first station load per session via get_station_storage(). No separate migration script needed; multi-tenant stations migrate independently on their own schedule.

5.3 Persistence

After seed_station_defaults() returns, the existing save_station_storage(station_id) path pushes changes to the JSON file or Postgres backing store. The migration is one in-memory pass plus one save — no separate transaction needed.

5.4 Failure modes & resilience

FailureOutcomeDetectionRecovery
Process killed mid-migration (disk full, OOM) Some nozzles migrated, others not. get_tank_id_for_nozzle falls back to pump.tank_id for unmigrated ones — no observable difference. Log entry for migration count is missing or partial Next station load completes the migration (idempotent guard skips already-set tank_ids)
Pump has no tank_id (malformed data) Nozzle.tank_id stays None; warning logged; fuel_type fallback handles routing Warning in logs Owner uses preset UI to set tank_id explicitly
Migration writes correct value but save fails In-memory state is correct, on-disk is stale. Next request reads stale state from disk on cold start. Save error in logs Migration re-runs on next load (idempotent), save retries

5.5 Rollback semantics

The migration is strictly additive — it only writes to a new field. Reverting the code would leave nozzle.tank_id as inert extra data. PumpStation.tank_id is never modified or removed. Zero risk of data loss.

12. Rollback Plan

Rollback triggers:

  • Critical export/report bug: row count or value mismatch > 0
  • Reconciliation number mismatch > 0.01 ZMW vs pre-deploy baseline
  • Shift creation failure rate > 1%
  • Owner-reported functional regression
  • Migration runs on > 1 station and produces unexpected logs

Rollback procedure:

  1. git revert the merge commit and redeploy. Zero schema changes means no DB rollback step needed.
  2. nozzle.tank_id data already written stays in the JSON / DB row but is unused by the reverted code. Inert.
  3. pump_station.tank_id was never modified — still authoritative for reverted code path.
  4. The migration that ran is idempotent: running again post-revert doesn't double-write.
  5. Communicate to operators: "Mixed-fuel feature temporarily withdrawn; resume single-fuel preset workflow."

Why this is safe: the refactor is strictly additive at the data layer. No fields removed, no schemas migrated. Reverting is cheap.
