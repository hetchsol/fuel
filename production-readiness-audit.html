1. Showstopper Issues

These issues make the system fundamentally unsuitable for production use. They must be resolved before any other fixes are meaningful.

#16 Showstopper Ephemeral Filesystem on Render Free Tier

The entire data storage uses in-memory Python dicts + JSON files on disk (backend/storage/stations/). On Render free tier, the service sleeps after 15 minutes of inactivity and wakes from a fresh deploy. The filesystem is ephemeral — every restart/sleep wipes all JSON files.

All station data, readings, shifts, sales, and tank levels are destroyed on every restart cycle.

Impact: Even with a perfect setup wizard, the owner would have to re-run it every time the server wakes up. Fundamentally incompatible with production use.
Fix Options:
A) Upgrade to Render paid tier (persistent disk) ~$7/month
B) Add an external database (Render free PostgreSQL, or Supabase/Neon free tier)
C) Use an external file store (S3, R2, etc.) for JSON persistence
#17 Showstopper Authentication Is Not Production-Grade
  • backend/app/api/v1/auth.py lines 15–97

Critical authentication flaws:

  • Users hardcoded in Python — no user registration, no user table
  • Passwords use simple SHA-256 (no salt, no bcrypt)
  • Tokens are predictable: token-{username}-{user_id} — anyone can forge tokens
  • No token expiry, no refresh tokens
  • No way to create/delete users at runtime
Impact: Any user who knows the token format can impersonate any other user. No ability to manage staff accounts in production.
#18 Showstopper No Data Persistence Layer
  • backend/app/database/storage.py line 59

STORAGE = _make_empty_storage() — everything lives in a Python dict. station_files.py handles JSON file read/write for some data, but not all data is persisted to disk on every write.

Impact: If the process crashes mid-operation, data can be lost or corrupted. No transactions, no locking, no backup mechanism.

2. Critical Issues

These issues will cause crashes, data corruption, or security failures in production.

#1 Critical Hardcoded Tank IDs in Backend Config
  • backend/app/config.py lines 68–69 — TANK_ID_DIESEL = "TANK-DIESEL", TANK_ID_PETROL = "TANK-PETROL"
  • backend/app/database/seed_defaults.py lines 39–44, 98–114, 269–274
Impact: These constants are imported and used throughout the backend. The helper function get_tank_id() (lines 156–175) returns these based on fuel type. Cannot configure different tank IDs without code changes.
#2 Critical Hardcoded Station ID (ST001) Throughout
  • backend/app/database/stations_registry.py lines 26–34 — auto-seeds ST001
  • backend/app/main.py lines 36, 45–46 — loads ST001 for backward compat
  • backend/app/api/v1/auth.py line 217 — get_station_context() defaults to "ST001"
Impact: Every API endpoint depends on ST001. If no station exists, all requests silently target a nonexistent station.
#3 Critical Hardcoded Island/Nozzle IDs
  • backend/app/database/seed_defaults.py lines 38–87 — seeds ISL-001 through ISL-006 with PS-001 through PS-006 and nozzle IDs ISL1-A/B through ISL6-A/B
  • backend/app/database/seed_defaults.py lines 258–275 — migration logic checks for exact island IDs
Impact: Complete island/pump/nozzle tree is seeded with hardcoded identifiers. All nozzle operations reference these IDs.
#4 Critical Hardcoded Account IDs
  • backend/app/database/seed_defaults.py lines 120–135 — seeds 15 default accounts (ACC-POS, ACC-GENSET, ACC-KAFUBU, etc.)
Impact: All accounts are created with specific IDs. Any code referencing accounts by ID will fail if seeds are removed.
#5 Critical POST /sales Crash Path
  • backend/app/api/v1/sales.py lines 89, 94–96

Uses storage['tanks'] directly. Iterates to find tank by fuel_type. If storage['tanks'] is missing, throws KeyError. If no tank matches fuel_type, target_tank stays None (line 96) but code then attempts to access it (line 99).

Impact: Production crash path — guaranteed 500 error when no tanks exist or fuel type doesn't match.
#6 Critical All Test Users Hardcoded to ST001
  • backend/app/api/v1/auth.py lines 22, 30, 47, 63, 71, 79, 95

7+ hardcoded users (user1, supervisor1, owner1, staff members) all with "station_id": "ST001". No user registration, no user database table.

Impact: If ST001 is removed or renamed, all test users become invalid. No mechanism to add or remove staff in production.
#7 Critical ST001 Backward Compatibility in main.py
  • backend/app/main.py line 45

Explicitly calls get_station_storage("ST001") on startup for backward compatibility. This creates a hard dependency on ST001 existing at application startup.

Impact: Application startup depends on ST001. Removing ST001 may prevent the application from starting.

#9 High Island-to-Tank Hardcoded Mapping
  • backend/app/api/v1/islands.py line 181

Maps product type to tank: "TANK-PETROL" if product == "Petrol" else "TANK-DIESEL". Assumes these exact tank IDs exist.

Impact: Breaks if tank IDs are different from the hardcoded values. No support for additional fuel types.
#10 High Dashboard Tank Cards Break with Missing Tanks
  • frontend/pages/index.tsx lines 32–33, 87–108

Finds tanks by tanks?.find((t: any) => t.fuel_type === 'Diesel'). If no Diesel tank exists, dieselTank is undefined; the TankCard component receives an undefined object, breaking UI rendering.

Impact: Dashboard crashes or renders broken UI when expected tanks don't exist.
#11 High Inventory/Services May Assume Account Existence
  • backend/app/services/inventory.py (if exists)

Likely has functions like process_credit_sale() that work with account IDs and may assume certain accounts (e.g., ACC-POS) exist.

Impact: Operations referencing specific account IDs will fail if seed accounts are removed.
#12 High Island Migration Logic Hardcoded
  • backend/app/database/seed_defaults.py lines 258–275

Migration logic assumes islands "ISL-001" through "ISL-006" exist in exact form and assigns product types to them. Breaks if island IDs differ.

Impact: Data migration will fail or produce incorrect results for non-default island configurations.

8. Architectural Concerns

8.1 Storage Architecture

  • Everything lives in Python dicts in memory (storage.py:59 — STORAGE = _make_empty_storage())
  • JSON files on disk for some persistence, but not all writes are immediate
  • Render free tier's ephemeral filesystem means all data is lost on restart/sleep (every 15 min of inactivity)
  • No transactions, no locking, no backup mechanism
  • No data integrity guarantees if process crashes mid-operation

8.2 Authentication Architecture

  • Users hardcoded in Python source code — no database table
  • SHA-256 password hashing with no salt (vulnerable to rainbow tables)
  • Tokens are predictable/forgeable: token-{username}-{user_id}
  • No token expiry or refresh token mechanism
  • No runtime user management (create/delete/update users)
  • No password reset capability

8.3 Persistence Architecture

  • station_files.py handles JSON file read/write for some data
  • Not all data is persisted to disk on every write operation
  • No write-ahead log or journaling
  • Corrupt JSON files = total data loss for that entity
  • No backup/restore capability

10. Production Acceptance Criteria

The system should not be considered production-ready until all of the following are met:

  1. Every storage['key'] direct access must use .get('key', {}) or .get('key', [])
    Prevents 500 errors from missing storage keys across ~30+ locations
  2. Frontend pages must handle empty tank/island arrays gracefully
    Dashboard, tank readings, and infrastructure pages must render without crashing when data is empty
  3. Test users must be removed or have no hardcoded station_id
    No hardcoded users in source code; implement proper user database with bcrypt password hashing
  4. API endpoints must NOT auto-seed on first access
    Force explicit configuration through a setup wizard; no implicit data creation
  5. Dashboard must not hardcode "TANK-DIESEL" / "TANK-PETROL" lookups
    Tank display must be fully dynamic based on API-returned data
  6. Authentication tokens must be cryptographically secure
    Use UUID4 or JWT with proper signing; implement token expiry and refresh
  7. Data must survive server restarts
    Implement external database (PostgreSQL) or persistent storage solution
  8. Reconciliation must use runtime fuel prices
    Use resolve_fuel_price() from settings instead of config constants

Bottom Line

The 15 code-level mitigations address the seeded data problem, but the application has 3 architectural gaps (storage, auth, persistence) that make it unsuitable for production regardless of code-level fixes. The recommended path is to first resolve the showstopper issues (external database + proper auth), then apply the code-level mitigations in priority order.
