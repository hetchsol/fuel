LPG Reconciliation — Current System vs. Proposed Document

Analysis of: LPG Recon.docx (owner's requested LPG workflow)
Against: Current Fuel Management app (backend/app/api/v1/lpg_daily.py, attendant_handover.py, models/models.py)
Date: 2026-04-22
Bottom line: The data model mostly matches. The money math on upgrades is wrong, upgrade revenue is excluded from the shift total, the handover flow cannot capture upgrades at all, and the empty-cylinder balance formula has a sign bug. Three of these silently understate revenue today.
Critical issues: 3 High: 3 Medium: 2 Low: 1 Fully correct as-is: 5

1. What the document asks for

The owner's LPG Recon.docx describes how a fuel-station LPG reconciliation spreadsheet should behave. Three transaction types and two calculated columns.

1.1 Three transaction types

TypeCustomer behaviourCharge the customerEmpty-cylinder effect
Refill Brings their own empty, gets it refilled. Size × RatePerKg Empty[size] +1
New Issue Buys a brand-new filled cylinder (no trade-in). Size × RatePerKg + Deposit[size] 0 (nothing returned)
Upgrade Brings a smaller empty, walks away with a larger filled cylinder. NewSize × RatePerKg + (Deposit[New] − Deposit[Old]) Empty[Old] +1, Empty[New] −1

1.2 Two calculated columns

EmptyCylinderChange =
  IF(TransactionType = "Refill",   +1,
  IF(TransactionType = "NewIssue",  0,
  IF(TransactionType = "Upgrade",
     IF(Size = OldSize, +1,
     IF(Size = NewSize, −1, 0)),
  0)))

RunningEmptyBalance = OpeningBalance + SUM(EmptyCylinderChange up to current row)

In plain English: the empty-cylinder count must update per transaction, by size, with a running total the attendant can see after every line.

2. What the system can do today

Every file path below is anchored to C:\Projects\Fuel\.

2.1 Data model — mostly present

Models in backend/app/models/models.py:

  • LPGCylinderShiftRow (lines 689–703) — per-size, per-shift row with opening_balance, opening_empty, receipts, traded_in, traded_out, sold_refill, sold_with_cylinder, closing_empty, and derived balance, value_refill, value_with_cylinder, total_value.
  • LPGCylinderTrade (lines 681–687) — one upgrade/downgrade line with from_size_kg, to_size_kg, quantity, derived price_difference and trade_type.
  • LPGDailyEntryInput/Output (lines 705–733) — shift-level payload with a trades array and a total_trade_revenue.
  • LPGStockLineItem (lines 838–879) — handover snapshot equivalent: opening_full, opening_empty, closing_full, closing_empty, sold_refill, sold_with_cylinder, damaged, variance_note.

2.2 Pricing — configurable

lpg_daily.py:34–43, 94–109, 114–162. Pricing is stored in lpg_pricing.json as price_refill and price_full_cylinder per size for the six sizes (3, 6, 9, 19, 45, 48 kg). Deposit is implicit: deposit = price_full_cylinder − price_refill. A supervisor-only PUT /lpg-daily/pricing endpoint exists. A backward-compatibility path for older price_per_kg + deposits format is also present.

2.3 Three transaction types — two correct, one miscalculated

DocumentApp implementationCorrect?
Refill → Size × RatePerKg value_refill = price_refill × sold_refill (lpg_daily.py:291) Yes
New Issue → Gas + Deposit value_with_cyl = price_with_cylinder × sold_with_cylinder (lpg_daily.py:292) Yes
Upgrade → NewSize × RatePerKg + (Deposit[New] − Deposit[Old]) price_diff = price_refill[to] − price_refill[from] (lpg_daily.py:266) No — see §3.1

2.4 Empty cylinder tracking — fields exist, logic manual

  • opening_empty / closing_empty stored per size per shift.
  • Previous-shift carry-forward works: GET /lpg-daily/previous-shift returns cylinder_empty_balances keyed by size (lpg_daily.py:216–221).
  • But closing_empty is attendant-entered. The app does not compute an expected closing-empty from refills + upgrades and does not flag variance on empties (variance check is filled-stock only, attendant_handover.py:234–235).

2.5 Shift integration — wired

  • LPG revenue feeds shift reconciliation via lpg_revenue on ShiftReconciliation (models.py:414–429, endpoint at reconciliation.py:74–133).
  • Recent commit 1e683c9 adds assigned_lpg to AttendantAssignment so a shift can be flagged as an LPG shift at creation.
  • Handover snapshot auto-seeds the daily LPG entry via _feed_daily_entries (attendant_handover.py:394+).

3. Differences — side-by-side

3.1 Upgrade pricing (CRITICAL)

CRITICAL   The app undercharges every upgrade transaction.

Document formulaApp formula (lpg_daily.py:266)
Charge NewSize × rate + (Deposit[New] − Deposit[Old]) price_refill[to] − price_refill[from]
3kg → 6kg example
(rate≈33/kg; deposit 3kg=330, 6kg=500)		 200 + (500 − 330) = 370 ZMW 200 − 147 = 53 ZMW
Shortfall per upgrade317 ZMW undercharged with default prices.

Root cause: The app treats an upgrade as "pay the gas-price difference" and omits the deposit difference entirely. The customer is walking out with a physically larger cylinder — that gap in deposit liability has to be settled.

Fix:

from_deposit = from_pricing['price_with_cylinder'] - from_pricing['price_refill']
to_deposit   = to_pricing['price_with_cylinder']   - to_pricing['price_refill']
price_diff   = to_pricing['price_refill'] + (to_deposit - from_deposit)

3.2 Upgrade revenue excluded from the grand total (CRITICAL)

CRITICAL   Shift reconciliation is missing upgrade revenue.

In lpg_daily.py:281–310, per-row total_value = value_refill + value_with_cylinder. grand_total_value (line 336) sums those rows. total_trade_revenue is computed at line 271 and stored separately on the output (line 344) — but never added into grand_total_value. Downstream reconciliation pulls grand_total, so upgrade revenue is not counted toward the shift's expected cash.

Fix: add grand_total += total_trade_revenue before assembling the output, or publish a grand_total_with_trades and switch reconciliation to that field.

3.3 Handover flow cannot record upgrades at all (CRITICAL)

CRITICAL   Attendants have no way to record an upgrade in the shift handover.

  • _process_stock_snapshot computes lpg_sales = value_refill + value_with_cyl only (attendant_handover.py:237–239). No trade handling.
  • _feed_daily_entries hard-codes "traded_in": 0, "traded_out": 0 when seeding the daily entry from the snapshot (attendant_handover.py:424).

Practical consequence: an upgrade has to be entered twice — once on the handover (as if it were a refill of the wrong size) and once via a separate POST /lpg-daily/entry with a trades array — or it is lost. Nobody will do that reliably.

Fix: extend StockSnapshotInput with a trades: List[LPGCylinderTrade] field, extract the trade-processing block out of lpg_daily.py:262–279 into a shared helper, call it from _process_stock_snapshot, and propagate traded_in/traded_out through _feed_daily_entries.

3.4 Balance formula double-counts returned empties (HIGH)

HIGH   Returned upgrade empties inflate the filled-stock balance.

Current formula (lpg_daily.py:290):

balance = opening_balance + receipts + traded_in
        - sold_refill - sold_with_cylinder - traded_out

traded_in is the empty the customer handed back; adding it to the filled balance overstates filled stock by one unit per upgrade on the from-size. traded_out is correctly subtracted (the station did give away a filled larger cylinder).

Fix:

balance = opening_balance + receipts
        - sold_refill - sold_with_cylinder - traded_out

expected_closing_empty = opening_empty + sold_refill + traded_in - traded_out_new_fill

Move traded_in out of the filled-balance line and into the expected-empty line (see §3.5).

3.5 Empty-cylinder tally is manual, with no variance check (HIGH)

HIGH   Empty balances are attendant-entered with no cross-check.

Document requires automatic per-transaction adjustment (§1.2). The app never derives an expected closing-empty, never flags a mismatch, and only variances the filled side (attendant_handover.py:234–235, 242–243).

Fix:

  1. Compute expected_closing_empty[size] = opening_empty[size] + sold_refill[size] + traded_in[size] − traded_out_from_new_fill[size].
  2. Flag variance exactly like the filled-stock check (attendant_handover.py:242–243), requiring a variance_note when non-zero.
  3. Expose the expected value live on the attendant UI so data-entry errors are caught immediately.

3.6 Deposits are derived, not first-class config (MEDIUM)

MEDIUM   Deposit is inferred as price_full_cylinder − price_refill. The document assumes an explicit DepositTable. Problems:

  • Supervisors cannot change just the deposit without recomputing price_full_cylinder.
  • The upgrade-pricing UI can't display "deposit difference = X" cleanly.
  • Historical revaluation is awkward because deposit and gas price are entangled in one snapshot.

Fix: add an explicit deposit field to each entry in lpg_pricing.json's prices map, keep price_full_cylinder as a derived view for compatibility, and update PUT /lpg-daily/pricing validation (lpg_daily.py:147–153).

3.7 Handover seeding bypasses previous-shift carry-forward (MEDIUM)

MEDIUM   _feed_daily_entries (attendant_handover.py:394+) builds a daily entry from the snapshot's opening_* values directly. The GET /lpg-daily/previous-shift endpoint is never consulted. If the attendant keys a wrong opening value on the handover screen, it propagates into the daily entry without cross-check.

Fix: have _feed_daily_entries call the previous-shift lookup and warn if the snapshot opening values disagree.

3.8 No hard stop / escalation on negative stock (LOW)

LOW   Oversell warnings exist for filled balance (lpg_daily.py:312–316) but there is no equivalent for negative empties and no supervisor-acknowledgement gate. Consider adding both.

4. What the app already gets right

  • Data model covers all three transaction types — sold_refill, sold_with_cylinder, trades[].
  • Per-size pricing is configurable and role-gated to supervisors/owners.
  • Previous-shift carry-forward for both filled and empty balances.
  • Filled-stock variance detection with mandatory variance_note.
  • Shift-level integration — LPG revenue flows into ShiftReconciliation; LPG assignment exists at shift creation.

7. Recommended sequencing

  1. Day 1 (fixes 1, 2, 4): three small backend patches + unit tests. Revenue starts counting correctly immediately.
  2. Day 2–3 (fix 3): extend handover to accept trades; update attendant UI. Removes the double-entry footgun.
  3. Day 4 (fix 5): expected-empty calculation and variance flag on both backend and UI.
  4. Day 5 (fixes 6, 7, 8): explicit deposit config, carry-forward cross-check, negative-empty guard.

Want a patch set for steps 1–2 first? Those are the revenue-leak fixes and are small enough to ship on their own.
